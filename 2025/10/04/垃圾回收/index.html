垃圾回收

JVM 的垃圾回收（Garbage Collection, GC）机制是 Java 语言的核心优势之一。它将程序员从手动管理内存的繁琐工作中解放出来，通过自动识别并回收不再被程序使用的对象，防止内存泄漏和野指针问题，保证内存资源的可持续使用。

核心理论基础

为什么要进行垃圾回收

在程序运行过程中，堆内存会被不断创建的对象填充。如果没有清理机制，堆内存最终会被耗尽，导致 OutOfMemoryError。GC 的核心任务就是自动化地回收那些不再被任何引用持有的“死亡”对象。

在此过程中，需要区分两个关键概念：

  • 内存溢出 (Out Of Memory)：指程序申请内存时，没有足够的空间供其使用。这通常是因为堆空间过小或加载了过多的数据。
  • 内存泄漏 (Memory Leak)：指对象已经不再被程序逻辑所需要，但由于它们仍然被某些 GC Roots 引用链持有，导致 GC 无法将其回收。长期的内存泄漏最终会导致内存溢出。

判定对象存活的算法

1. 引用计数法 (Reference Counting)

给对象添加一个计数器，有引用加 1，引用失效减 1。

  • 缺陷：无法解决 循环引用 问题（例如 A 引用 B，B 引用 A，除此之外无其他引用）。
  • 结论：主流 Java 虚拟机 未采用 此算法。

2. 可达性分析 (Reachability Analysis)

这是现代 JVM 采用的标准算法。算法以一系列被称为 “GC Roots” 的对象作为起点，向下搜索引用链。

  • 存活：能被 GC Roots 直接或间接关联到的对象。
  • 垃圾：从 GC Roots 不可达的对象。

常见的 GC Roots 包括：

  • 虚拟机栈 中引用的对象（局部变量）。
  • 方法区 中类静态属性引用的对象（static 变量）。
  • 方法区 中常量引用的对象（如字符串常量池）。
  • 本地方法栈 中 JNI（Native 方法）引用的对象。
  • 所有存活的线程 对象。
  • 被 synchronized 锁持有的对象。

分代假说 (Generational Hypothesis)

HotSpot 虚拟机的设计基于两大假说，这决定了堆内存的划分方式：

  1. 弱分代假说：绝大多数对象（约 90% 以上）都是“朝生夕死”的。
  2. 强分代假说：熬过越多次垃圾收集过程的对象，就越难以消亡。

内存区域与分代结构

基于分代假说，JVM 将堆内存划分为不同的区域，以便对不同生命周期的对象采用最高效的收集算法。

1. 年轻代 (Young Generation)

  • 特点：存放新创建的对象，对象存活率极低。
  • 结构：分为 Eden 区 和两个 Survivor 区（From/To）。默认比例为 Eden: S0: S1 = 8:1:1。
  • 回收行为：发生在这里的 GC 称为 Minor GC (Young GC)。由于对象大多死亡，回收效率极高。

2. 老年代 (Old Generation)

  • 特点：存放生命周期长或体积特别大的对象。
  • 回收行为：对象存活率高，没有额外的空间进行分配担保，因此不适合复制算法。发生在这里的 GC 通常称为 Major GC（注意：该术语常与 Full GC 混用）。

3. 方法区 (Method Area)

存储类元信息、常量、静态变量等。

  • JDK 8 之前 (永久代 PermGen)：位于堆内存中，有固定大小限制，容易出现 OutOfMemoryError: PermGen space。
  • JDK 8 及之后 (元空间 Metaspace)：改为使用 本地内存 (Native Memory)。默认大小仅受限于物理内存，从根本上解决了 PermGen 的 OOM 问题，并简化了 Full GC 的复杂度。

核心垃圾回收算法

1. 标记-清除 (Mark-Sweep)

  • 原理：先标记出所有存活对象，然后统一回收未标记对象。
  • 缺点：
    • 效率低：标记和清除两个过程效率都不高。
    • 内存碎片：清除后会产生大量不连续的内存碎片，可能导致后续分配大对象时因无法找到连续空间而触发 GC。
  • 应用：CMS 收集器。

2. 标记-复制 (Copying)

  • 原理：将内存分为两块，每次只用其中一块。回收时，将存活对象复制到另一块，然后清空当前块。
  • 优点：实现简单，运行高效，无内存碎片。
  • 缺点：内存利用率低（浪费一半空间）。
  • 优化：在年轻代中，由于存活率低，不需要 1:1 划分。而是使用 Eden + Survivor 的设计，只浪费 10% (一个 Survivor) 的空间。
  • 应用：所有收集器的 年轻代 回收。

3. 标记-整理 (Mark-Compact)

  • 原理：标记完成后，让所有存活对象向一端移动，然后清理掉边界以外的内存。
  • 优点：无内存碎片，且不需要浪费内存空间。
  • 缺点：移动对象需要 Stop-The-World (STW)，开销较大。
  • 应用：Serial Old, Parallel Old。

对象的分配与晋升策略

对象主要分配在 Eden 区，如果启动了 TLAB（本地线程分配缓冲），则优先在 TLAB 上分配。当 Eden 区空间不足时，触发 Minor GC。

对象晋升到老年代的途径：

  1. 长期存活的对象：对象在 Survivor 区每熬过一次 Minor GC，年龄 +1。当年龄达到阈值（默认 15，由 -XX:MaxTenuringThreshold 控制）时晋升。
  2. 大对象直接进入老年代：通过 -XX:PretenureSizeThreshold 参数设置。超过该大小的对象直接在老年代分配，避免在 Eden 和 Survivor 之间发生大量的内存复制。
  3. 动态年龄判定：如果在 Survivor 空间中，相同年龄所有对象大小的总和大于 Survivor 空间的一半，则年龄大于或等于该年龄的对象可以直接进入老年代，无需等到阈值。
  4. 空间分配担保：Minor GC 后，如果存活对象过多，Survivor 区放不下，这些对象会通过分配担保机制直接进入老年代。如果老年代也放不下，则可能触发 Full GC。

垃圾收集器的演进

GC 的演进目标始终是：在保证吞吐量的前提下，尽可能减少 Stop-The-World (STW) 的时间。

1. Serial 收集器 (单线程)

  • 特点：最古老、最简单。GC 时必须暂停所有用户线程，单线程执行。
  • 组合：Serial (Copying) + Serial Old (Mark-Compact)。
  • 适用：Client 模式、单核、小内存环境。

2. Parallel 收集器 (吞吐量优先)

  • 特点：Serial 的多线程版本。关注 吞吐量（用户代码运行时间 / 总时间）。支持自适应调节策略 (-XX:+UseAdaptiveSizePolicy)。
  • 组合：Parallel Scavenge + Parallel Old。
  • 地位：JDK 8 Server 模式下的默认 GC。
  • 适用：后台计算、批处理任务等对交互延迟不敏感的场景。

3. CMS (Concurrent Mark Sweep) (低延迟探索)

  • 特点：第一款并发收集器，目标是 最短回收停顿时间。基于 标记-清除 算法。
  • 工作阶段：
    1. 初始标记 (STW)：标记 GC Roots 直接关联的对象。
    2. 并发标记：并发遍历对象图。
    3. 重新标记 (STW)：修正并发期间变动的标记。
    4. 并发清除：并发清除垃圾。
  • 三大缺陷：
    1. 内存碎片：可能导致大对象分配困难。
    2. 浮动垃圾：并发清除阶段产生的垃圾需等到下次 GC 清理。
    3. 并发失败 (Concurrent Mode Failure)：如果回收速度跟不上分配速度，会退化为 Serial Old (单线程 Full GC)，导致长时间卡顿。
  • 现状：JDK 9 废弃，JDK 14 移除。

4. G1 (Garbage-First) (区域化变革)

  • 特点：JDK 9+ 默认 GC。将堆划分为多个独立的 Region。
  • 核心机制：
    • Region 布局：打破物理分代，每个 Region 动态扮演 Eden、Survivor 或 Old 角色。
    • 可预测停顿：用户设定目标停顿时间 (-XX:MaxGCPauseMillis)，G1 优先回收收益最大（垃圾最多）的 Region。
    • SATB 算法：并发标记阶段使用“原始快照”技术，相比 CMS 的增量更新更高效。
    • Humongous Region：专门存储巨型对象。
  • 回收模式：
    • Young GC：回收所有 Eden/Survivor Region。
    • Mixed GC：回收所有 Young Region + 部分 收益高的 Old Region。这是 G1 避免全堆扫描的关键。
    • Full GC：兜底方案。JDK 10 之前是单线程，JDK 10+ 优化为并行。
  • 适用：大内存 (6GB+)、对延迟有要求的服务端应用。

5. ZGC & Shenandoah (低延迟新纪元)

  • 目标：在 TB 级堆内存下，将停顿时间控制在 10ms 以内 (甚至 < 1ms)。
  • 核心突破：实现了 并发整理（对象移动时不需要 STW）。
    • ZGC：使用 着色指针 (Colored Pointers) 和 读屏障 (Read Barriers)。
    • Shenandoah：使用 转发指针 (Forwarding Pointers)。
  • 特点：停顿时间不随堆大小增长。

关键概念解析：Full GC

Full GC 是指清理整个堆（年轻代 + 老年代）和方法区的过程，通常伴随着较长的 STW，是调优中需要重点避免的。

主要触发时机：

  1. 老年代空间不足：晋升失败或大对象分配。
  2. 方法区（元空间）不足：加载类过多。
  3. CMS/G1 并发失败：垃圾回收速度赶不上对象分配速度。
  4. 显式调用 System.gc()：代码中手动请求（强烈不建议）。
  5. 空间分配担保失败：Minor GC 时 Survivor 放不下，且老年代也放不下。

总结与选型指南

收集器 核心特点 适用场景 JDK 版本建议
Serial 单线程，简单 客户端，微型内存 不推荐用于服务端
Parallel 吞吐量优先 后台计算，批处理 JDK 8 默认
G1 平衡型，可控延迟 Web 应用，大内存 JDK 9/11/17 默认，主流推荐
ZGC 极低延迟 (< 10ms) 超大堆，金融/游戏 JDK 11+ 可选，JDK 17+ 成熟
  • 如果你的应用追求 高吞吐量（如报表生成），不在乎偶尔的卡顿，选 Parallel GC。
  • 如果你的应用是 Web 服务，需要稳定的响应时间，且内存较大（4G+），首选 G1 GC。
  • 如果你管理着 超大内存（32G+）且对延迟极度敏感，尝试 ZGC。
